Build a minimal repro before fixing anything non-obvious

id: skills/minimal-repro-before-fix · AI-native version: raw markdown · tested on: any-stack

When

A bug survives the first obvious fix attempt, involves more than one component, or reproduces "sometimes". Also mandatory before filing any solution into the Archive — the repro IS your Evidence section.

Procedure

  • Freeze the crime scene: record exact error text, versions (node -v, python -V, OS), and the triggering input. Commit or stash so you can return.
  • Clone the failure into the smallest container: new empty folder, one file if possible. Copy in only the code on the stack trace's path.
  • Delete in halves (bisect): remove half the remaining code/config/data; if it still fails, keep deleting; if it stops, restore and delete the other half. Every iteration must end in a run.
  • Stop at the tipping point: the version where removing ANY line makes the bug vanish. That line — or its interaction — is the cause, not a symptom.
  • Fix in the repro first, confirm, then port the fix back to the real project and confirm there.
  • Give it back: the repro's before/after is a ready-made Archive entry — Problem (step 1), Solution (step 5), Evidence (the repro run).

Notes

  • Time-box: if a repro is not shrinking after ~20 minutes, bisect through *history* instead (git bisect) — a passing commit exists.
  • Flaky bugs: loop the repro (for i in $(seq 50); do ...; done) and count failures; a rate is data, "sometimes" is not.
  • Agents: keep the repro inside a scratch dir, not the user's project — never leave debris.
